## 모니터링

- **Railway 대시보드**: 로그 / 메트릭 / 배포 상태
- **수동 health check**: `curl https://<도메인>/api/v1/families -X POST`
  → 200 응답 받으면 OK
- (향후) Sentry / Better Stack 도입

## 사고 카탈로그

### Cold start 응답 5초+ 지연
**증상**: 첫 요청이 응답까지 5~10초 걸림. 이후는 빠름.
**원인**: Railway 무료 티어 — 일정 시간 트래픽 없으면 컨테이너 sleep.
**해결**: 정상 동작. 사용자에게는 로딩 표시로 안내.
**예방**: Railway 유료 ($5+/월) 전환 시 always-on 가능.

### 배포 후 500 에러 (DB 컬럼 누락)
**증상**: API 호출 시 SQL 에러 — 새 컬럼 INSERT/SELECT 실패.
**원인**: 코드는 새 컬럼 사용하는데 `@PostConstruct DDL`이 아직 실행 안 됨.
**해결**: 보통 첫 부팅 시 자동 ALTER. 안 되면 Railway 콘솔에서 수동:
```bash
railway connect postgres
\d bl_cry_samples       # 컬럼 확인
alter table bl_cry_samples add column if not exists pitch_mean_hz double precision;
```
**예방**: `DatabaseConfig.SchemaInitializer.init()` 의 ALTER 문이 모두 `if not exists` 인지 확인.

### DB 연결 timeout
**증상**: API 호출 시 `Connection timed out` 또는 hang.
**원인 1**: Railway PG 일시 장애.
**해결**: Railway status 페이지 확인. 보통 자동 복구.
**원인 2**: 연결 풀 고갈 (동시 사용자 급증).
**해결**: `application.yml` 의 datasource pool size 조정 (현재 기본값).

### 배포는 됐는데 새 코드 반영 안 됨
**증상**: GitHub push 했는데 Railway에 옛날 응답.
**원인**: 자동 빌드 트리거 누락 (드물게).
**해결**: Railway 대시보드 → 해당 서비스 → Deployments → "Trigger Deploy".

### 이전 버전으로 롤백 필요
**해결**: Deployments 탭 → 이전 successful 배포 → "Redeploy".

또는 git revert + push.

### 운영 DB에 잘못된 데이터
**시나리오**: 버그로 잘못된 라벨 / 음수 amount 등이 들어감.
**해결**:
```bash
railway connect postgres

# 진단
select * from bl_feed_records where amount_ml < 0 or amount_ml > 500;

# 정리 (백업 후!)
update bl_feed_records set amount_ml = 0 where id = '<id>';
```

영향 큰 변경은 PSQL `BEGIN; ... ROLLBACK;` 으로 dry-run 권장.

### 시크릿 유출 의심
**즉시**:
1. Railway → Variables → 해당 시크릿 회수/회전
2. PG 비밀번호 회전 후 ENV 갱신
3. `git log` 에 시크릿 들어갔는지 확인 — 들어갔으면 `git rm + force push` 또는 BFG repo cleaner
